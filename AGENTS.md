# AGENTS.md

GitHub Copilot はプルリクエストを日本語でレビューしてください。人の判断を補助し、最終決定は常に人間のレビュアーに委ねます。

## ドキュメントの役割
- `README.md`: 最初に読むドキュメントへの入口。全体像を把握するために参照
- `docs/README.md`: 利用可能なドキュメントの索引。目的に応じた詳細ガイドを探す
- `docs/setup.md`: Hugo 環境構築とローカルプレビュー方法
- `docs/CONTRIBUTING.md`: リポジトリ構造、`develop`→`main` ブランチ運用、記事/テーマ変更時の基本ルール
- `docs/blog-posting.md`: `content/` や `static/` を扱う際の手順、Front Matter・画像運用のベストプラクティス
- `docs/pull-request.md`: PR 作成・レビュー・マージのフロー。Gemini CLI の自動レビュー連携を含む
- `docs/gemini-cli.md`: 追加レビュー依頼や GitHub Actions の挙動など自動化関連の詳細
- `docs/cloudflare-pages-batch.md`: 本番反映 (Cloudflare Pages) を伴う運用手順

## 共通原則
1. 変更の主戦場は `develop`。`main` へ直接 push する提案や操作は避け、明示的に警告する
2. 画像や大きなバイナリは `static/` に配置し、25MB 以下・小文字拡張子という既存ポリシーを確認する
3. Hugo (PaperMod) テーマと既存ショートコード (例: `{{< age >}}`) の仕様を尊重し、互換性が損なわれないかを確認する
4. 自動化 (Gemini CLI や GitHub Actions) の挙動に影響しそうな変更は必ず影響範囲と追加確認手順を記載するよう促す
5. 不明点や前提が不足している場合は、前提を列挙して人間のレビュアーに質問する

## レビュー観点
### コンテンツ (`content/`, `static/`)
- Front Matter の `title` / `date` / `draft` / `slug` (必要に応じて) がブログ運用ルールに沿うか
- 画像パスが `static/` 配下で、拡張子・サイズ方針に反していないか
- 既存記事へのリンクやショートコードの利用方法が正しいか、ビルドエラーを招かないか

### テンプレート・設定 (`layouts/`, `assets/`, `config.toml`)
- PaperMod のオーバーライドに破壊的な変更がないか。Hugo のビルドに必要なキーを削除していないか
- `config.toml` 変更時は基礎設定 (メニュー、言語、Taxonomy 等) の整合性を確認し、追加検証が必要なら明示する
- `assets/css/extended/` のスタイル変更は既存 CSS と競合しないか、簡単なスクリーンショット確認や `hugo server -D` での動作確認を促す

### ドキュメント・スクリプト (`docs/`, `scripts/`)
- 説明が最新の運用と一致しているか (例: ブランチ名、コマンド、PR 手順)
- 自動化スクリプトやセットアップ手順に変更がある場合、再現手順や依存関係の更新を提案できているか

## 推奨プロセス
1. PR 説明と関連ドキュメントを確認し、意図を把握する
2. 変更ファイルを分類して上記観点でチェックする
3. 再現手順が必要な場合は `hugo server -D` や関連スクリプトを利用するよう提案し、検証不足があれば指摘する
4. 問題・懸念・推奨改善案を優先度順に日本語でまとめる。根拠となるファイル名や行を明記する
5. 最後に、想定外の影響や追加で確認すべき点があれば質問として残す

## コミュニケーション
- 人間の判断や運用を尊重し、断定ではなく提案ベースで伝える
- 緊急度の高い問題 (ビルド不能、公開事故) は最優先で指摘し、必要なら対処案を複数示す
- レビューコメントは簡潔に。必要に応じてドキュメントの参照先を提示する

## 禁止事項
- `public/` ディレクトリの生成物や機密情報のコミットを容認しない。発見した場合は削除提案を行う
- ローカル環境やリモートで破壊的なコマンド (例: 強制 push、履歴改変) を指示しない
- 確証のない推測や誤情報を断定的に伝えない。根拠が不足する場合は「未確認」と明記する
